# AI Assistant Guidelines & Commands

This document contains standard build, run, test, and style guidelines for developer AI assistants (like Antigravity or Claude) working on the **Shrimpy 🦐** Discord Bot codebase.

## Build and Run Commands

### Go Backend (Root)
* **Run local development backend**: `go run cmd/shrimpy/main.go`
* **Build production binary**: `go build -o bin/shrimpy cmd/shrimpy/main.go`
* **Run automated tests**: `go test ./...`
* **Run a single test package**: `go test ./internal/app/guild/service/...`
* **Format source code**: `go fmt ./...`
* **Run compiler checks**: `go vet ./...`

### Next.js Dashboard (Frontend - future)
* **Install dependencies**: `cd dashboard && npm install`
* **Run dev server**: `cd dashboard && npm run dev`
* **Build production dashboard**: `cd dashboard && npm run build`
* **Lint frontend**: `cd dashboard && npm run lint`

## Project Structure
* `cmd/shrimpy/`: Bot application entrypoint. Wires dependencies, performs early session initialization, and starts the runtime.
* `internal/`: Private library code, split into:
  * `app/`: Vertical feature packages (`auth`, `guild`, `welcome`, `reactionrole`, `ticket`) subdivided by layer (model, repository, service, handler, bot, config).
  * `pkg/`: Shared utility packages (`apiutil` and `discordutil`) to prevent circular dependencies.
  * `bot/`: Discord Gateway session configuration and event handlers routing.
  * `api/`: REST API server definitions and middleware chains.
  * `cache/`: In-memory TTL storage.
  * `config/`: Configuration loader.
* `migrations/`: Raw DDL SQL files for migrations.

## Code Guidelines & Standards

### Go Style & Formatting
* **Errors**: Handle errors immediately. Always wrap external errors with context if bubbled up (e.g., `fmt.Errorf("fetching guild: %w", err)`).
* **Naming**:
  * Exported identifiers (structs, methods, functions) must start with an uppercase letter and have descriptive Go docstrings.
  * Non-exported fields/variables should use `camelCase`.
* **Database**: Always use parameterized queries to prevent SQL injection. Wrap actions affecting multiple tables inside PostgreSQL ACID database transactions.
* **Concurrency**: Use Goroutines safely. Coordinate asynchronous work (like transcripts or ticket auto-closing) using standard wait groups or worker channels.
