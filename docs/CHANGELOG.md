## [Unreleased]
### Added
- **Emoji picker (unicode + custom server emoji)** across the dashboard: a shared `EmojiPicker` component with a "Standard" tab (curated, searchable unicode set) and a "Server" tab listing the guild's own custom emoji, wired into the **Ticket Panels** (per-category button emoji) and **Reaction Roles** (mapping emoji) screens; an `EmojiView` helper renders custom emoji as CDN images everywhere they're previewed. New `GET …/guilds/:id/discord/emojis` endpoint (served from the gateway `State` cache, so Discord's `GUILD_EMOJIS_UPDATE` events keep it fresh without a REST call; frontend adds a 5-min TTL cache). Shared `discordutil` helpers (`ParseComponentEmoji`, `ReactionEmojiAPIName`, `CustomEmojiMention`/`URL`) normalize the three emoji representations Discord needs — button/select `ComponentEmoji`, reaction-endpoint `name:id`, and the picker's canonical `<:name:id>`/`<a:name:id>` mention form — so custom emoji render on ticket buttons and reaction-role reactions grant/revoke correctly. Also fixes reaction-role mappings never being persisted from the dashboard in non-demo mode (the create flow now calls the per-emoji endpoint).
- **Emoji in free-text fields**: an insert-at-cursor `EmojiInsertButton` (compact smiley trigger sharing the same combined unicode + server-emoji panel) is wired into the message-content and embed-**description** fields where Discord actually renders custom emoji — **Welcome** DM body, and **Ticket Panels** panel content/embed description + per-category opening content/description. Emoji are spliced at the caret (replacing any selection) rather than overwriting the field, and the panel live previews now render inserted custom emoji as images via a new `EmojiText` helper. Titles/author/footer are intentionally excluded since custom emoji don't render there.
- **Ticket "Resolve" status** ([TECHNICAL_SPEC.md](file:///d:/Pesronal/Projects/Discord%20Bot/docs/v1/TECHNICAL_SPEC.md) §6, §4.5; [COMMAND_REFERENCE.md](file:///d:/Pesronal/Projects/Discord%20Bot/docs/v1/COMMAND_REFERENCE.md)): a new `resolved` ticket status, distinct from `closed` — staff mark a ticket as handled via a `Resolve` button (alongside the existing Claim/Close) or `/ticket resolve`, without locking the channel or generating a transcript. Resolving (re)starts an auto-close timer using the category's existing auto-close window, so an unanswered resolved ticket closes itself; un-resolving reverts to Open/Claimed and clears the timer. New `POST …/tickets/:id/resolve` and `.../unresolve` endpoints, `ticket_status` enum gains `resolved` (migration `008_ticket_resolved_status`), and the dashboard Tickets page gets a matching filter tab, badge, and row actions.

### Changed
- **Resolved all 8 open questions in USER_JOURNEY.md §14** (now retitled "Decisions Log"), and propagated each into the spec:
  1. **Demo mode** gated behind an explicit `/demo` route; the mock-fallback is removed from `lib/api.ts` for real sessions — no silent mock rendering on API failure.
  2. **Ticket detail** is a full route (`/tickets/[id]`), not a drawer.
  3. **Sidebar regroup:** CONFIGURE → **"Server Management"** (Ticket Panels, Welcome incl. auto-roles, Reaction Roles); **Settings** (General, Staff & Access) stays a separate group for bot/server-level plumbing. Bot-wide status/error-logs across every server it's in is noted as a future `/admin/apps` extension, out of scope for now.
  4. **Staff (Level 2)** sidebar: Server Management + Settings groups are **fully hidden**, not read-only.
  5. **Overview stats:** counts + avg. resolution only for v1; sparkline/chart deferred to "Dashboard v2."
  6. **Invite permissions:** scoped permission integer (View/Manage Channels, Manage Roles, Manage Threads, Send Messages, Embed Links, Read Message History, Add Reactions, Use External Emojis, Manage Messages) replaces `permissions=8` (Administrator).
  7. **Ticket detail conversation is read-only in v1** — no dashboard reply composer, no `POST …/tickets/:id/messages` endpoint. Staff reply in the Discord thread.
  8. **Participants (`S-05`) deferred** — the `ticket_participants` table and its 3 endpoints are dropped from v1; per-category support roles (`ticket_categories`, multi-role) already auto-grant every role-holder access to a ticket's thread, covering the real workflow.
  
  Net effect on [Appendix B](file:///d:/Pesronal/Projects/Discord%20Bot/docs/v1/USER_JOURNEY.md#appendix-b--data-contract--endpoint-coverage): the journey's new backend surface shrinks from "6 endpoints + 1 table" to **4 endpoints + 0 new tables**. [TECHNICAL_SPEC.md](file:///d:/Pesronal/Projects/Discord%20Bot/docs/v1/TECHNICAL_SPEC.md) §3.2 and §4.4 updated to drop `ticket_participants` and the `messages`/`participants` routes accordingly.

## [1.1.0] - 2026-06-21
### Added
- **`bot_settings` singleton table** (`migrations/002_bot_settings.up.sql`): Stores all four Discord credentials (`DISCORD_TOKEN`, `DISCORD_CLIENT_ID`, `DISCORD_CLIENT_SECRET`, `DISCORD_REDIRECT_URI`) encrypted at rest with AES-256-GCM. Eliminates the need to update Railway environment variables when credentials change.
- **`internal/app/settings/` vertical module**: Full model / repository / service / handler stack following the established vertical module pattern.
- **Admin REST API endpoints** (`GET`, `PUT /api/v1/admin/settings`, `POST /api/v1/admin/settings/reconnect`): Dashboard-facing CRUD for bot credentials. Token and client secret are always masked in GET responses.
- **Live bot reconnect** (`bot.Reconnect()`): Updating the token via `PUT /api/v1/admin/settings` automatically closes and reopens the Discord Gateway connection on the same `*discordgo.Session` pointer — no restart required, all handlers preserved.
- **`internal/api/middleware/admin.go`**: Admin-only middleware that checks `managed_guilds` JWT claim or `OWNER_DISCORD_ID` env var override.
- **`OWNER_DISCORD_ID` env var**: Optional; grants a specific Discord user unconditional access to the admin settings endpoints.
- **First-boot seeding**: On startup, if `bot_settings` is empty and `DISCORD_*` env vars are set, credentials are automatically encrypted and persisted to the DB. After first boot, env vars can be safely removed from Railway.

### Changed
- **`internal/config/config.go`**: All four `DISCORD_*` environment variables changed from required (`mustGetEnv`) to optional (`getEnv`). Added `HasDiscordSeed()` helper and `getEnvFallback()` for Railway's `PORT` precedence.
- **`cmd/shrimpy/main.go`**: Startup sequence reordered — `bot_settings` table migrated first, credentials seeded/loaded from DB, Discord session constructed from DB token, then remaining modules built.
- **`internal/app/auth/handler/handler.go`**: Auth callback now fetches OAuth2 credentials from the settings service (30s cached DB lookup) instead of from startup config.
- **`internal/api/server.go`**: Admin route group added under `/api/v1/admin/`.
- **`railway.toml`**: `healthcheckTimeout` increased from 10s to 30s to accommodate cold-start DB connection + migration time.
- **Technical Spec** ([TECHNICAL_SPEC.md](file:///d:/Pesronal/Projects/Discord Bot/docs/v1/TECHNICAL_SPEC.md)): Updated sections 3 (schema), 4 (API), 8 (directory structure), 9 (config), 10.2 (Railway vars), 10.4 (Vercel vars), 10.5 (docker-compose).

### Fixed
- **Railway health check failure**: HTTP server now starts before the Discord Gateway connection so `/health` is reachable during Railway's health check window.
- **Railway `PORT` env var**: Config now reads Railway's injected `PORT` variable first, falling back to `API_PORT`, then `8080`.
